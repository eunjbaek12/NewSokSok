# 클라우드 동기화

## 개요

Google 계정으로 로그인한 사용자의 단어장 데이터를 클라우드(PostgreSQL)에 자동으로 동기화한다. 모든 데이터 변경(단어 추가/삭제, 암기 상태 변경 등)이 발생하면 2초의 디바운스 후 전체 데이터를 서버에 업로드한다. 게스트 모드는 동기화 없이 로컬 SQLite에만 저장된다. 현재 구현은 "마지막 쓰기 우선(Last Write Wins)" 방식이며, 멀티 디바이스 병합 로직은 아직 구현되지 않았다.

## 관련 파일

- `contexts/VocabContext.tsx` — `syncToCloud()`, `debouncedSync()`, `loadCloudData()` 구현
- `server/auth.ts` — `GET /api/sync/data`, `POST /api/sync/data` API 엔드포인트
- `shared/schema.ts` — `cloud_users`, `cloud_vocab_data` PostgreSQL 테이블
- `lib/vocab-storage.ts` — 로컬 SQLite 데이터 소스

## 주요 개념

| 개념 | 설명 |
|------|------|
| `cloud_users` | PostgreSQL 테이블 — Google 사용자 정보 (id UUID, google_id, email, display_name, avatar_url) |
| `cloud_vocab_data` | PostgreSQL 테이블 — `user_id(FK)`, `data_json JSONB` — 전체 단어장 배열을 JSONB로 저장 |
| `x-user-id` | 동기화 API 인증 헤더 — `user.id` (UUID, cloud_users.id) 값 |
| 디바운스 2초 | 연속적인 변경에서 마지막 변경 후 2초가 지나야 업로드 실행 |
| 전체 교체 방식 | 동기화 시 기존 클라우드 데이터를 덮어씀 (부분 업데이트 없음) |

## 동작 시나리오

### 시나리오 1: Google 로그인 후 초기 데이터 로드

1. Google 로그인 완료 후 `VocabProvider`의 `useEffect`가 실행된다.
2. 먼저 `initSeedDataIfEmpty()`로 로컬 DB를 초기화한다.
3. `authMode === 'google'`이면 `loadCloudData()` 호출:

> **`loadCloudData()` 로직**:
> ```
> GET /api/sync/data (헤더: x-user-id: userId)
> ↓
> 응답의 lists 배열이 비어있으면?
>   → 로컬 데이터를 POST /api/sync/data 로 업로드 (초기 마이그레이션)
> 응답의 lists 배열에 데이터가 있으면?
>   → "Phase 2 pending" 로그만 출력 (현재 병합 로직 미구현)
> ```

4. `loadCloudData()` 완료 후 `refreshData()` 호출로 로컬 SQLite에서 최신 데이터 로드.

---

### 시나리오 2: 데이터 변경 시 자동 동기화

1. 사용자가 단어를 추가/삭제하거나 암기 상태를 변경한다.
2. `VocabContext`의 모든 mutation 함수들은 작업 완료 후 `debouncedSync()`를 호출한다:
   ```
   addWord() → Storage.addWord() → refreshData() → debouncedSync()
   deleteWords() → Storage.deleteWords() → refreshData() → debouncedSync()
   toggleMemorized() → Storage.toggleMemorized() → refreshData() → debouncedSync()
   ... (모든 mutation 동일 패턴)
   ```
3. `debouncedSync()` 내부:
   - 기존 타이머가 있으면 `clearTimeout`으로 취소
   - 2초 후 `syncToCloud()` 실행

> **`syncToCloud()` 로직**:
> ```
> authMode !== 'google' 이면 즉시 return (게스트 모드)
> ↓
> Storage.getLists() 로 최신 로컬 데이터 조회
> ↓
> POST /api/sync/data
> Header: x-user-id: user.id, Content-Type: application/json
> Body: { lists: [...] }
> ↓
> 실패해도 console.warn만 (사용자에게 에러 표시 없음)
> ```

---

### 시나리오 3: 서버 동기화 저장 처리

서버 `POST /api/sync/data` (`server/auth.ts:114`):

1. 헤더에서 `x-user-id` 추출 → 없으면 401.
2. body의 `lists` 배열 유효성 확인 → 배열이 아니면 400.
3. PostgreSQL UPSERT 실행:
   ```sql
   INSERT INTO cloud_vocab_data (user_id, data_json, updated_at)
   VALUES ($1, $2::jsonb, NOW())
   ON CONFLICT (user_id)
   DO UPDATE SET data_json = $2::jsonb, updated_at = NOW()
   ```
4. 성공 시 `{ success: true }` 응답.

---

### 시나리오 4: 2초 디바운스 동작 예시

사용자가 빠르게 단어 3개를 연속 추가하는 경우:

```
t=0ms: 단어 A 추가 → debouncedSync() → 타이머 2000ms 설정
t=500ms: 단어 B 추가 → debouncedSync() → 기존 타이머 취소, 새 타이머 2000ms
t=800ms: 단어 C 추가 → debouncedSync() → 기존 타이머 취소, 새 타이머 2000ms
t=2800ms: syncToCloud() 실행 → A, B, C 모두 포함된 전체 데이터 1회 업로드
```

---

### 시나리오 5: 게스트 → Google 전환 시나리오 (현재 한계)

1. 게스트 모드에서 단어장을 만들고 단어를 추가한다 (로컬만 저장).
2. 로그아웃 후 Google로 재로그인한다.
3. `loadCloudData()` 실행 시:
   - 신규 사용자면 cloud_vocab_data가 `[]` → 로컬 데이터를 클라우드에 업로드 ✓
   - 기존 Google 사용자이고 클라우드에 데이터가 있으면 → "Phase 2 pending" 로그만 출력, **로컬 데이터와 병합 없음**

> **현재 한계**: 멀티 디바이스에서 각각 수정한 데이터를 병합하는 로직이 미구현 상태. `data_json`은 항상 마지막으로 `POST /api/sync/data`를 호출한 디바이스의 데이터로 덮어씀.

## 예외 / 엣지 케이스

- **네트워크 오류**: `syncToCloud()` 실패 시 `console.warn`만 출력, 사용자 알림 없음. 로컬 SQLite 데이터는 안전하게 보존됨
- **대용량 데이터**: 단어장이 많을수록 전체 데이터를 매번 전송하므로 페이로드가 커짐 (부분 업데이트 미구현)
- **동기화 확인 불가**: 현재 클라이언트에서 마지막 동기화 시간이나 성공 여부를 UI에 표시하지 않음
- **토큰 만료**: 서버에서 Google 토큰 재검증 없이 `user.id`만으로 인증. 로컬 저장된 사용자 ID를 신뢰
